**Data:** 2026-07-16
**Decisão:** Regra de cobrança de hora extra de proteções e adicionais (`tipo_preco='per_day'`) virou configurável por item no painel admin, em vez de fixa no código. Novo campo `regra_hora_extra` (`proporcional` | `integral_apos_tolerancia`) em `protecoes`/`adicionais` (`sql/034_regra_hora_extra_protecoes_adicionais.sql`, default `proporcional` — comportamento histórico preservado para todo item já cadastrado). `proporcional` é a regra que já existia (`calcDias()`: tolerância de 1h, fração em quartos de diária entre 1h–4h, diária cheia acima de 4h). `integral_apos_tolerancia` é a nova opção: mesma tolerância de 1h, mas ao ultrapassar cobra 1 diária **inteira** extra, sem fração. Implementada como nova função `calcDiasItem(retirada, devolucao, regraHoraExtra)` na fonte canônica (`supabase/functions/_shared/pricing.js`) e replicada nas 3 cópias sincronizadas (`apps/site/shared/`, `apps/intake-admin/shared/`, `extensions/cotacao-rapida/shared/` — via `tests/pricing-parity.test.js`). A categoria do veículo nunca usa `calcDiasItem` — continua sempre com `calcDias()` (proporcional), a mudança é só para proteções/adicionais, como pedido.
**Justificativa:** Pedido do usuário: poder trocar a política de hora extra de um item específico direto no painel, sem depender de deploy de código. Confirmado o desenho (schema + motor de preço + admin + deploy) antes de implementar.
**Impacto:** **Schema:** `sql/034_regra_hora_extra_protecoes_adicionais.sql` (aplicado em produção via MCP). **Motor de preço:** `calcDiasItem()` adicionada em `_shared/pricing.js` + 3 cópias; `tests/pricing.test.js` ganhou 4 testes cobrindo a nova função. **Servidor** (`supabase/functions/criar-solicitacao/index.ts`, deploy v21): proteção e cada adicional agora calculam sua própria diária via `calcDiasItem(data_retirada, data_devolucao, item.regra_hora_extra)`, em vez de reusar a diária global da categoria. **Client** (`apps/site/js/pricing-adapter.js`): novo `calcBaseProtecao()` centraliza o cálculo da proteção (antes duplicado inline em `summary.js`/`step4.js`×2/`mobile.js`, sempre usando a diária global — bug latente corrigido de passagem); `calcSubtotal()` (usado por todos os adicionais) agora calcula a diária por-item internamente. **Admin** (`apps/intake-admin/pages/protecoes.js`/`adicionais.js`): novo campo select "Cobrança de Hora Extra", visível só quando `Tipo de Cobrança = Por dia` (toggle ao trocar o tipo, mesmo padrão do toggle de estrangeiro do site). **Visualizador de reservas do admin** (`reservas.js`, `verReserva`/`imprimirReserva`): também corrigido para não recalcular com a diária global. **Cotação rápida** (`extensions/cotacao-rapida/sidebar.js`): novo `getDiasItem()`, mesmo padrão, para as cotações não ficarem erradas para itens com a regra nova. `tests/site-pricing-adapter.test.js` ganhou testes para `calcSubtotal`/`calcBaseProtecao` com as duas regras. Validado ponta-a-ponta: teste real contra a Edge Function de produção (registro de teste removido depois) confirmou `159,98 = 0,5 diária proporcional da categoria + 1 diária integral do adicional` — exatamente o esperado; teste no browser do site confirmou o mesmo comportamento na sidebar. `npm test` 94/94. `get_logs` (edge-function) sem erros novos pós-deploy.

**Data:** 2026-07-14
**Decisão:** Revisão de segurança RLS (último item da lista de consultoria) encontrou e corrigiu **duas vulnerabilidades críticas de escalonamento de privilégio**, não relacionadas a `apps/site` diretamente mas no mesmo projeto Supabase (`lxfnqzuzohudqwibgdic`): (1) a policy de `UPDATE` em `public.usuarios` só validava `id = auth.uid()`, sem restringir quais colunas podiam mudar — **qualquer usuário autenticado (mesmo "cliente") conseguia se autopromover a admin** com um único `UPDATE` direto via REST API (`.update({role:'admin'}).eq('id', auth.uid())`), sem precisar de nenhuma outra permissão. (2) O trigger `fn_criar_usuario_no_signup()` (dispara em `auth.users` insert, `SECURITY DEFINER` — bypassa RLS) confiava cegamente no `role` vindo dos metadados que o próprio usuário informa no cadastro (`raw_user_meta_data->>'role'`). Combinado com o `tenant_id` da Igufoz ser público (hardcoded em `apps/site/supabase.js:7`, visível a qualquer um via DevTools), **um visitante anônimo conseguia se cadastrar já como admin do tenant** via a API pública de signup do Supabase Auth — sem nunca ter passado pelo site. Corrigido com `sql/033_bloquear_auto_promocao_admin.sql`: trigger `fn_impedir_auto_promocao_usuario()` (BEFORE UPDATE em `usuarios`) bloqueia qualquer mudança de `role`/`tenant_id` feita por quem não é admin; `fn_criar_usuario_no_signup()` reescrita para sempre gravar `role='cliente'`, ignorando o valor informado no cadastro — promoção a admin só é possível pela policy existente de INSERT feita por um admin já autenticado.
**Justificativa:** Último item da revisão de segurança pedida pelo usuário como consultor ("conferir se todas as tabelas públicas têm RLS restritivo"). A investigação foi além de checar `rls_enabled` (todas as tabelas do site já tinham RLS + policies corretas) e usou os advisors de segurança do Supabase (`get_advisors`) para achar as duas funções/trigger de auth com o problema real — que não é sobre RLS ausente, mas sobre lógica que **contorna** RLS (SECURITY DEFINER) ou uma policy existente que valida o dono da linha mas não o conteúdo da mudança.
**Impacto:** Migration aplicada em produção via MCP do Supabase (`sql/033_bloquear_auto_promocao_admin.sql`). Validado com testes dentro de transações `BEGIN`/`ROLLBACK` (nenhum dado alterado): tentativa de mudar `role` de um usuário existente → bloqueada pela exceção do trigger; `UPDATE` de campo normal (`whatsapp`) no mesmo usuário → passa normalmente, sem regressão. Re-rodado `get_advisors` (security): as duas funções novas não aparecem mais no lint de `search_path` mutável (ambas já nascem com `SET search_path = public`). **Itens de menor prioridade identificados mas não corrigidos** (fora do escopo desta revisão, aguardando decisão): bucket de storage `categorias` permite listagem pública (baixa sensibilidade — só fotos de veículo); ~15 funções com `search_path` mutável pré-existentes (hardening, não exploração conhecida); proteção contra senha vazada desativada no Auth (`HaveIBeenPwned`, afeta login de admin/operador, não o site público); extensões `pg_net`/`btree_gist` instaladas no schema `public` em vez de um schema dedicado.

**Data:** 2026-07-14
**Decisão:** Corrigido segundo bug (distinto do `goToStep()`, entrada seguinte deste log) por trás do mesmo report do usuário: recarregar a página (F5) em qualquer step sobrescrevia silenciosamente `S.retData`/`S.devData` sempre que sobrassem as chaves `qs_retData`/`qs_devData`/`qs_local` em `sessionStorage` — pré-preenchimento de uma busca rápida feita na landing (`index.html`), que só deveria valer para *iniciar* uma reserva do zero. `loadSession()` aplicava esse pré-preenchimento de forma incondicional, sem checar se já existia um rascunho (`igufoz_rascunho`) em andamento — então um reload no Step 4, por exemplo, mantinha categoria/proteção/adicionais intactos mas trocava as datas de retirada/devolução pelas da busca rápida (comportamento relatado pelo usuário: "a data é alterada... as demais informações se mantêm"). Pior: `S.dias` (e portanto o valor de diárias cobrado) não era recalculado, ficando descolado do novo período exibido. Corrigido: o pré-preenchimento de `qs_*` só se aplica quando **não** existe rascunho salvo (`!raw`, sessão nova de verdade); em qualquer caso as chaves `qs_*` são sempre limpas após o load, para não sobrarem à espreita num reload futuro. Adicionado também `calcDias()` logo após restaurar a sessão no boot, para nunca confiar cegamente em `S.dias` persistido.
**Justificativa:** Reportado pelo usuário: "o erro ocorre quando a página é atualizada... estou no step 4... atualizo a página. A data é alterada... as demais informações se mantêm." Reproduzido exatamente: sessão completa até Step 4, com `qs_retData`/`qs_devData` remanescentes em `sessionStorage`, reload troca as datas e desalinha o valor cobrado das diárias exibidas.
**Impacto:** `script.js`: `loadSession()` (guarda `!raw`, limpeza incondicional das chaves `qs_*`), boot (`calcDias()` após restaurar sessão). Validado no browser: (1) sessão completa até Step 4 + `qs_retData`/`qs_devData` residuais + reload → datas e diárias permanecem exatamente como estavam antes do reload, chaves `qs_*` removidas; (2) fluxo real landing→reserva (sessão nova, sem rascunho) + `qs_retData`/`qs_devData`/`qs_local` → pré-preenchimento continua funcionando normalmente. Sem erros de console. `npm test` 60/60, inalterado.

**Data:** 2026-07-14
**Decisão:** Corrigido bug crítico introduzido pela steps-bar clicável (entrada anterior deste log, mesmo dia): `goToStep(n)` pulava direto para qualquer etapa já alcançada (`S.maxStep`) sem nenhuma revalidação — diferente de `nextStep()`, que sempre roda `validate()`. Isso permitia o seguinte fluxo: cliente chega ao Step 4 normalmente (período válido) → clica no chip "Período" para voltar e ajustar algo → edita data/hora para uma combinação inconsistente (ex.: devolução 05:30 no mesmo dia da retirada 09:00, hora de devolução anterior à retirada) → clica direto no chip "Confirmação" → cai no Step 4 com o resumo mostrando um período inválido (visto pelo usuário: "15/07–15/07", retirada 09:00, devolução 05:30, mas ainda exibindo 8 diárias de uma edição anterior) e, mais grave, poderia enviar a solicitação nesse estado. `goToStep()` agora recalcula `calcDias()` e reconfere `dias > 0` + `retLocal`/`devLocal`/`catId` sempre que o destino é > Step 1; se inválido, força a permanência no Step 1 e chama `validate()` para destacar o campo com erro, em vez de deixar o cliente avançar.
**Justificativa:** Reportado pelo usuário com screenshot do Step 4 mostrando um resumo inconsistente (mesma data de ida/volta, hora de devolução antes da retirada, mas ainda com 8 diárias cobradas de um estado anterior). Root cause: a navegação por clique nos chips (feature do mesmo dia) não tinha o mesmo gate de validação que `nextStep()` sempre teve.
**Impacto:** `script.js`: `goToStep()`. Validado no browser reproduzindo o cenário exato do report: Step 4 (válido) → chip "Período" → editar devData para o mesmo dia + devHora 05:30 (antes de retHora 09:00) → clicar chip "Confirmação" diretamente → bloqueado, permanece no Step 1 com `devHora` marcado `input-error` e mensagem de campo visível. Cenário de controle (dados válidos) confirmado navegando normalmente pelos chips sem bloqueio. Sem erros de console. `npm test` 60/60, inalterado. **Lição:** ao adicionar um atalho de navegação que contorna o fluxo linear normal (`nextStep()`→`validate()`), replicar explicitamente o mesmo gate de validação no atalho — não presumir que dados de uma visita anterior a uma etapa continuam válidos.
